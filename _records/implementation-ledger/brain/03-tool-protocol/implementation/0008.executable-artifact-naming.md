## Implemented

- **The law**: one-artifact-per-file exports mirror the file stem exactly, role word included (Regime 2 — like `logMemoryEventTool`, `logMemoryEventSchema`, `logMemoryEventPrompt`, `banNativeDatePolicy`). Behavior functions (helpers, repositories) stay action-first with no role tail (Regime 1). Executables are artifacts and join Regime 2.
- Converted all three executables from `function` declarations to `const` arrow artifacts mirroring their files:
  - `log.memory.event.executable.ts` → `logMemoryEventExecutable` (was `runLogMemoryEvent`)
  - `write.user.memory.executable.ts` → `writeUserMemoryExecutable` (was `writeUserMemoryExecute` — duct-tape tail)
  - `read.user.memory.executable.ts` → `readUserMemoryExecutable` (was `readUserMemoryExecute`)
- Updated all callers: three tool files + `memory.tool.test.ts`. AI-facing tool name `log_memory_event` unchanged (spec-owned string).
- New lexicon slice `_lexicon/backend/executable.lexicon.constant.ts`: `executable` (role, backend scope), wired into the backend aggregate during the human unlock ritual.
- Amended `run` in the global actions lexicon: **reserved for orchestration-level execution only** (daemons, guard runs, loop dispatch) — never domain behavior.

## Rules Captured

- Artifact exports mirror file stems exactly; collisions with repository behavior names become impossible by construction.
- `run` is orchestration-reserved vocabulary.
- No new tails are ever invented per incident (`Execute` was exactly that).

## Deferred
