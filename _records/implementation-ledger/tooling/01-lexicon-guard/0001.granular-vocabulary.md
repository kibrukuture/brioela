# 0001 - Granular Vocabulary

## Scope

## Status

done

## Implemented

- Split the lexicon guard vocabulary out of three large files into granular owned slices.
- Kept the stable aggregate exports: `globalLexicon`, `backendLexicon`, `productLexicon`, and added `toolsLexicon`.
- Added focused lexicon folders for `global`, `backend`, `product`, and `tools` ownership.
- Kept the loader simple: it still consumes aggregate arrays, but the arrays are assembled from small domain files.
- Made Brain migration manifest tooling inherit Brain/product vocabulary because that tool is Brain-migration-owned.
- Removed the broad padded vocabulary bucket by splitting it into exact origin and status slices.

## Rules Captured

- Lexicon growth must happen by adding a focused owned slice, not by swelling one large file.
- Tool vocabulary is not automatically global vocabulary.
- Product vocabulary can be used by product-owned tooling only when scope resolution proves that ownership.
- Padding bucket names are not allowed, even inside guard internals.
- Public aggregate names stay stable so guard loading does not churn when vocabulary is reorganized.

## Not Implemented

- Automatic generation of lexicon slice indexes.
- Per-slice owner metadata inside `LexiconWord`.

## Next

2. Add a guard rule that requires new lexicon files to live under an ownership folder.
3. Consider a small lexicon index generator only if manual slice exports become noisy.
