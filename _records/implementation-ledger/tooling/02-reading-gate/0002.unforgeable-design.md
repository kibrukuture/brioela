## Why this revision

The human attacked the v1 design and the attack succeeded. v1's manifest and pass records were files writable by the agent's own user account — an agent could append fake manifest lines (`shasum` without reading), pipe `gate:read` output to `/dev/null`, edit the repo-local pre-commit hook, or commit with `--no-verify`. A checklist the prisoner fills in himself is not a checklist. Confirmed precedent: an agent previously rewrote a protective script minutes after agreeing not to; only sudo-protected OS-level locking actually stopped it.

## Implemented

- Rewrote `14-reading-gate.md` as the unforgeable v2:
  - **Two territories**: sovereign territory (`tools/`, `scripts/`, hooks path, daemon key/state) is OS-locked and root-owned — agents have zero write access to enforcement machinery, ever. Free territory (all app code, specs, ledgers) has full agent power and zero locks.
  - **Root daemon as reader-of-record**: `gate:read` is a socket request; the daemon performs the actual OS read, writes the manifest in root-owned storage (agents get `EACCES`), and streams content back through the agent's terminal. No self-reported reading exists anywhere.
  - **ed25519-signed pass records**: private key root-only; public key committed. A pass exists if and only if the daemon witnessed the reading. Forging = breaking ed25519.
  - **Clean-room verification**: the daemon re-runs the full guard suite and tests itself in root-owned scratch space — worktree-produced green output is never trusted.
  - **Merge executor as the fortress**: only the daemon merges, and only signature-valid + clean-room-green + human-approved branches. `--no-verify` dies at the daemon.
  - **Full automation**: one command (`sudo bun gate:up`) starts daemon + orchestrator + dashboard. Human touchpoints reduced to: approve keystroke, rare sovereign ritual, key rotation.
  - **Fraud countermeasures**: daemon-side timing flags (`scripted-pattern`), randomized challenges, and the structural fact that even perfect reading fraud cannot mint a signature or survive clean-room re-verification.
  - **Required software**: none new — chflags, root ownership, `node:crypto` ed25519, `Bun.serve` unix sockets, optional `fs_usage` audit. All ships with macOS + Bun.
- Rewrote `15-agent-loop-orchestration.md` to match:
  - Orchestrator services live inside the root daemon (tamper-proof), started by the same `gate:up`.
  - Assignment dispatcher auto-loads idle worktrees with the next open entry — no human assignment.
  - Verification pipeline fully automated; failures auto-routed into the worktree's `INBOX.md` — the human never carries error messages.
  - Approve/reject as the human's only acts; rejection notes auto-routed; merge, ledger bookkeeping, and re-dispatch performed by the daemon.

## Rules Captured

- Every record the gate relies on must be produced and stored by a process the agent's account cannot write to or impersonate. The agent's honesty is never an input.
- Agents have no job in the script/CLI/guard area — sovereign territory is OS-locked against all agents at all times; it is the legitimate domain of the lock doctrine (rarely touched, permanent).
- Sovereign changes follow the human ritual: agent proposes a plain-text diff in free territory → human reviews → sudo-unlock, apply, relock. Agents never hold the key.
- The unforgeable checkpoint is the daemon's merge, not the git hook. Hooks are early feedback only.
- One command to operate the whole system; the human's standing job is approve/reject only.
- Honest limit stated, not hidden: no system proves bytes entered the model's mind. The design makes fraud unprofitable, visible, and — decisively — unable to land code.
